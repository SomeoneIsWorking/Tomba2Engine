USER 2026-07-22: pressing P to pause while fps60 is ON shows a BLACK SCREEN. At 30fps pause presumably holds the last frame correctly (user reports this as an fps60-specific fault). MECHANISM TO CHECK FIRST: fps60 presents by re-running the world pass one frame behind under lerped input chokes into an isolated sink, then merging with verbatim 2D (see the fps60 entry in docs/findings/render.md). A PAUSE stops the game advancing, so the interpolator's two endpoints stop being refreshed - if the paused present keeps consuming the interp path it will lerp between a stale pair, or between a valid frame and an EMPTY sink, and an empty sink presents as black. So the suspect is the pause path presenting through the interpolated pass instead of re-presenting the last REAL frame. Related known code: the debug/pause loop calls PadInput::pumpHostInput rather than serviceFrame precisely so the pad clock does not advance while frozen (external/psxport/runtime/recomp/native_boot.cpp) - check whether the fps60 present is correspondingly frozen, or whether it keeps running with nothing feeding it. Repro: run windowed or drive the pause via the debug server, toggle fps60 on, pause, shot. Fix must NOT be an anchor/stamp special case (banned by the CLAUDE.md lerp clause) - the correct behaviour is that a paused frame is simply the last real frame presented through the same path.

**2026-07-22:** 2026-07-22 MECHANISM ESTABLISHED by code reading (operator), fix not yet implemented. The debug-pause loop in external/psxport/runtime/recomp/native_boot.cpp (~line 421) keeps the window alive by calling GpuState::gpu_repaint() every iteration. gpu_repaint -> present_window() -> blit_src(s_vram, s_disp_x, s_disp_y) -> gpu_vk_present(core, src, ...) — i.e. it re-presents the PSX VRAM FRONT BUFFER. That is the psx_render picture. Under pc_render the picture is not in s_vram at all: it is produced by the VK geometry rasterizer through GpuState::gpu_present_ex / gpu_fps60_present_pass (gpu_native.cpp:1276/1423). So pausing re-presents VRAM, which under pc_render is not the frame — hence black. PREDICTION TO TEST FIRST, because it changes the card's title: this should reproduce at 30 fps too, i.e. it is a pc_render bug and not an fps60 bug; the user simply hit it at 60. Verify before fixing. THE FIX SHOULD USE WHAT NOW EXISTS: as of 2026-07-22 both presents go through Fps60::presentPass(core, t) and the real frame IS presentPass(core, 1.0f), so 're-present the current frame without advancing logic' has a natural single expression — re-run the same pass, or re-present the VK target it filled. Do NOT add a third present path; that is the two-code-paths bug (#17) which was just removed. NOTE the comment on gpu_repaint claims 'the VK readback reflects exactly what is on screen (vkshot reads the same region this presents)' — if that is wrong under pc_render then vkshot while paused is also lying, which would matter for every paused-state investigation. HARNESS WARNING, cost me several attempts: driving a pause headlessly is awkward. PSXPORT_REPL=1 with no stdin blocks the game so the debug server times out; without REPL the frame loop runs to completion and exits. Use PSXPORT_DEBUG_SERVER=1 and keep the process alive properly, then dbg commands 'pause' / 'vkshot <path>' / 'frame' on 127.0.0.1:5959.
