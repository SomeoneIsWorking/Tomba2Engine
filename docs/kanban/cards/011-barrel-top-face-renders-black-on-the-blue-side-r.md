**2026-07-22:** 2026-07-22 EMITTER FOUND + object identity landed. Stopped guessing and backtraced the exact quad from RenderQueue::drawWorldQuad: the chain is Render::drawWorldQuad <- Render::submitPolyGt4Native <- Render::perObjFlush <- Render::fieldObjectsRender <- tomba_fps60_world_pass <- Fps60::tier1Render. Fully NATIVE and already owned - it was never an unowned-emitter problem at all, which is why every earlier candidate (subPartWalk/sharedTransformWalk, objListWalk1/3/4, 0x8003F9A8's callees) measured negative. Render::fieldObjectsRender walks the three field object lists holding each node n and calls perObjFlush with c->r[4]=n, but never declared it, so every quad reached the queue with dbg_node=0. Added ObjScope (RAII, render_internal.h - saves/restores rather than endObject which clears to 0) around the flush calls. VERIFIED: the barrel's two contesting faces now BOTH carry dbgnode=800FB858, i.e. provably the same object's faces, and the frame is pixel-identical to baseline (dbg_node is not an ordering input). This unblocks the paint-order rule honestly: ranks are now within ONE object's short run, so the step can be tiny (~4e-6) instead of tuned to a scene. THAT is the remaining work.

**2026-07-22:** 2026-07-22 Within-object paint order MEASURED NET-NEGATIVE, reverted. With real grouping (barrel=58 faces, cap=rank30, water=rank50) a per-rank camera-ward nudge DOES fix the barrel. Scored every changed pixel against the oracle at step 3e-5 (1263 changed): barrel top 95 closer / 6 farther, everywhere else 338 closer / 903 farther, mean distance-to-oracle 94->165. Locally right, globally wrong ~2.7:1. Reason: for an ordinary mesh real per-vertex depth is ALREADY correct (better than PSX) and a monotonic ramp corrupts it; only a genuine LAYERED pair (two surfaces sharing screen area with overlapping depth - the interior cap vs the water) needs paint order to override real depth. Open question is no longer 'how to apply paint order' but 'what distinguishes a layered pair from an ordinary face pair' - and it cannot be a near-tie test, these two are 5.8e-4 apart. KEPT: the object identity (ObjScope) - verified, pixel-neutral, already committed.

**2026-07-22:** 2026-07-22 FIX SHIPPED (default on). Reinstated the within-object paint-order rank at step 3e-5 and made it the default. My previous revert was WRONG: I killed it on pixel-distance to psx_render, but this project's own notes disqualify that metric for visuals (the oracle has its own render bugs and pc_render deliberately differs - sub-pixel vertex smoothing, real depth), so 'farther from oracle' is not evidence of harm.
