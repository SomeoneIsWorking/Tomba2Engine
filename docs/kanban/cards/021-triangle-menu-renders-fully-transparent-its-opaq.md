**2026-07-22:** 2026-07-22 FIXED, and it CORRECTS the previous note on this card. The claim that 'the renderpsx toggle is BLIND to this one — pc and psx legs are pixel-identical (0 diff)' is FALSE: the default exec leg with PSXPORT_RENDER_PSX=1 renders the whole menu correctly (scratch/screenshots/uilayer/menu_pcexec_psxrender.png). The fault was never upstream of the renderer split — it was purely a missing pc_render producer. Root cause from guest data, no picture needed: the menu is drawn by controller FUN_800346BC through the two shared UI leaves FUN_8007E1B8 (POLY_FT4 groups) and FUN_8007E6DC (SPRT groups); PSXPORT_PRIMAT=160,120,323 shows one opaque prim (op=2D semi=0 raw=1 tp=384,0 clut=480,249 bbox 28,51-292,180) on the psx leg and NOTHING on the pc leg, and debug otattr attributes 195 op-0x2D packets to fn=0x8007E2F8 under caller=0x800346BC. Fix = new class PauseMenu (game/ui/pause_menu.cpp): a SCOPED leaf tap that runs each gen body untouched and re-derives the quads host-side at RQ_OVERLAY, plus the menu's own two full-screen guest quads (opaque black backdrop; subtractive 0x404040 dim at bucket 132). The non-obvious part: paint order is the guest's OT BUCKET (descending, LIFO within a bucket), not call order — call order paints the black panel over the item icons and the divider. Render::emitUiFt4 also had to walk its group entries back-to-front (0 px field-HUD regression). RESULT: 0 differing pixels of 76800 vs the psx_render leg, which matches the user reference element for element. Dialog box unchanged (bucket-softlock f1200). Diagnostics: debug pausemenu.
